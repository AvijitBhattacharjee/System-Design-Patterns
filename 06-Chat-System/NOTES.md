4. Interview Q&A
Q: How do you route a message from Alice (Server 1) to Bob (Server 2)?
A: When Bob connects, Server 2 writes user:bob → server2_id into Redis. When Alice sends a message, Server 1 looks up Bob's server, then either pushes directly over an internal gRPC channel between servers, or publishes to Kafka and Server 2 consumes it. The Kafka approach is preferred because it survives server crashes.
Q: What if Alice and Bob are on the same server?
A: The routing lookup shows they share a server — no Kafka hop needed. Server delivers in-memory and persists to Cassandra. This is the fast path.
Q: How do you estimate scale?
A: WhatsApp handles ~100B messages/day. That's ~1.15M messages/second. With Cassandra (each node handles ~50k writes/s), you need ~24 Cassandra nodes. For chat servers, each WebSocket holds ~10KB; a 16GB server can hold ~1.5M connections. Back-of-napkin: 500M users, 10% online = 50M connections = ~35 chat servers.
Q: How do you handle message history when a user gets a new device?
A: The client sends GET /messages?channel_id=X&before=<timestamp>&limit=50. Your API server queries Cassandra with a range scan on the clustering key. Paginate using the oldest message_id from the previous page as the cursor.
Q: Where does end-to-end encryption fit?
A: E2E encryption (Signal protocol) means the server never sees plaintext. Messages are encrypted client-side with the recipient's public key. The server stores and forwards ciphertext. Key exchange happens out-of-band (a key server stores public keys). Tradeoff: server-side search and moderation become impossible.
Q: How do you handle thundering herd on reconnect after a server crash?
A: Clients reconnect with exponential backoff with jitter — e.g., wait = min(cap, base * 2^attempt) + random(0, base). This spreads the reconnect storm over several seconds instead of hammering the load balancer simultaneously.
